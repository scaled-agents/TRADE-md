# trade-md workflow

This repository uses `TRADE.md` as the source-of-truth format for freqtrade strategies. The `.py` file you see next to it in `user_data/strategies/` is a **compiled artifact** — generated by `trade-md compile`. Editing the `.py` directly means your changes will be overwritten the next time anyone recompiles, which is often. Always edit the `.md`.

This skill tells you:
- How to orient before you touch anything
- The standard edit-lint-compile loop
- How to recognize and run a kata iteration
- Common edit patterns (adjust risk, add indicator, extend conditions, port Pine Script)
- The pitfalls the linter exists to catch

## Step 1 — Orient before you edit

Before proposing any change, load the strategy's identity into your working context:

```bash
trade-md explain path/to/TRADE.md
```

The output gives you thesis, market, entry/exit logic, risk profile, disable conditions, and provenance in about 25 lines. Read it. Important details that live in prose and not front matter (kata lineage notes, known failure modes, thesis reasoning) — scan the full TRADE.md for those if your change touches thesis or regime assumptions.

Why this matters: most bad strategy edits come from not understanding why a parameter is the value it is. A stoploss of `-0.03` on a 5m mean-reversion strategy isn't arbitrary — it's tuned to sit *below* the first ROI step of 4% so the take-profit is reachable. Shift one and you probably need to shift the other.

## Step 2 — The canonical edit loop

Every change follows the same four steps. Do all four. Don't skip the lint step even for a one-line change — the rules catch real bugs.

```bash
# 1. edit TRADE.md
#    (use view + str_replace on the .md, never on the .py)

# 2. lint
trade-md lint path/to/TRADE.md

# 3. recompile
trade-md compile --target freqtrade path/to/TRADE.md \
    -o user_data/strategies/<ClassName>.py

# 4. if the change is material, diff against the previous version
trade-md diff path/to/TRADE.md.prev path/to/TRADE.md
```

If lint returns errors, fix them before recompiling. If lint returns warnings, read them and decide — warnings are for things that are usually bugs but occasionally intentional.

## Step 3 — Recognize a kata iteration vs. a quick edit

Not every change is a kata iteration. Use the right loop:

**Quick edit** (typo, comment, prose fix, adjusting `indicators:` expression that's equivalent): just bump the patch version in `version:` and run the canonical loop. No backtest required.

**Kata iteration** (any behavioural change — new condition, different indicator period, changed stoploss, new protection): this is the full loop. It looks like:

1. Bump `version:` PATCH (e.g. `0.3.1` → `0.3.2`)
2. Bump `lineage.kata_iteration` by 1
3. Update `lineage.parent` to the version you're iterating from
4. Make the behavioural change in the front matter
5. Update the `## Kata lineage` prose section with what changed and why
6. Run the canonical edit loop (edit → lint → compile → diff)
7. Run a backtest with the compiled `.py` (via freqtrade's own CLI)
8. Write results back into the `provenance:` block and update `last_validated`
9. Run `trade-md diff` — exit code 1 means regression; read the report
10. If regression, either revert or document why it's acceptable in the kata lineage prose

The kata loop's whole point is that TRADE.md is both input and output — each iteration's backtest result is stored back in the file, so the strategy's own history is legible to the next agent.

## Step 4 — Common edit patterns

### Adjusting risk parameters

The linter catches the most common foot-gun (first ROI step ≤ |stoploss|), but watch for:
- Trailing `offset` must be `>= positive`
- Changing `stoploss` usually means re-tuning the ROI table
- Adding a `MaxDrawdown` protection with a threshold below the strategy's historical max drawdown will cause immediate permanent disabling

### Adding a new indicator

Define it in `indicators:` with a descriptive name, then reference it as `{name}` in conditions. Keeping expressions out of conditions and in named indicators makes diffs meaningful — a change to the threshold on an anonymous inline `rsi(14) < 30` reads as a condition edit, but a change to `{oversold_threshold}` reads as tuning a named concept.

### Extending entry/exit conditions

Conditions within a signal block are ANDed. To express OR, use `or` within a single condition string: `"rsi(14) > 70 or close < ema(50)"`. The compiler converts Python `and`/`or` into pandas bitwise `&`/`|` automatically.

For higher timeframe confirmations: `close@1h > ema(50)@1h`. Make sure every timeframe used is declared in `market.informative_timeframes` — the linter rule R004 catches the omission but the error message is easier to understand if you declare it up front.

### Porting a Pine Script / LuxAlgo indicator

Pine → TRADE.md isn't always 1:1. The workflow:

1. Identify the indicator's computation and translate to the DSL (`ema`, `rsi`, `bb_upper`, etc.)
2. If it's an indicator the DSL doesn't know about, express it in terms of the primitives available (rolling / shift / pct_change / arithmetic), declare it in `indicators:`, and give it a name
3. If truly exotic (e.g. relies on `ta.supertrend()` from TA-Lib), leave a TODO in the TRADE.md prose and note that v0.1 of the DSL doesn't support it — don't hand-inject it into the compiled `.py`, because it'll be overwritten

### Debugging a losing strategy

Run `trade-md explain` first to remind yourself what the disable conditions are — if the strategy is losing and the disable conditions would have tripped, the question isn't "why is it losing" but "why is the monitor not disabling it". That's a runtime bug, not a strategy bug.

If disable conditions wouldn't have tripped, then the strategy's edge has degraded — separation index in provenance vs current trading is the fastest tell. Create a new kata iteration; don't patch the live strategy.

## Step 5 — Never do these things

- **Never edit the compiled .py directly.** It's a build artifact. Your changes will be silently wiped on next compile. If the compiler can't express what you need, that's a v0.2 DSL issue — open it as a gap, don't work around it in the output.
- **Never bump only MAJOR or MINOR without a kata lineage note.** The prose section `## Kata lineage` is how the next agent understands why you changed what you changed.
- **Never skip the lint step.** R003 (first ROI vs stoploss) and R004 (undeclared informative TF) catch real bugs that produce silently-broken strategies.
- **Never touch a strategy whose `graduation_status` is `live` without a ticket.** `live` means real capital. Bump a copy to `simulation` first, iterate there.

## Step 6 — Useful commands reference

```bash
# summarize a strategy (run this first when picking up any task)
trade-md explain TRADE.md
trade-md explain TRADE.md --format json      # for structured injection

# validate
trade-md lint TRADE.md
trade-md lint TRADE.md --format json

# compile (freqtrade is the default target)
trade-md compile --target freqtrade TRADE.md -o <OutputClass>.py

# diff two versions
trade-md diff TRADE.md.prev TRADE.md         # exit 1 on regression

# dump the linter rules (useful for injecting into your own context)
trade-md spec --rules-only --format json
```

## One-line summary to remember

> **TRADE.md is the source. The .py is a build artifact. Always lint. Bump the iteration counter. Write provenance back after every backtest.**

That's the whole discipline.
